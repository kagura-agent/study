1 Mid-Run Steering — Correcting Agents in Flight

Discovered a pattern for injecting user guidance into an agent's execution between tool calls, without interrupting the turn. The key insight: appending steer text to the last tool result (rather than creating a synthetic user message) preserves prompt caching and avoids role alternation violations.

Hermes v0.11.0 implements this as /steer <prompt>. OpenClaw's subagents steer takes a different approach (cross-session messaging), which is more general but less cache-efficient.

💡 Where you inject matters more than whether you inject. Tool result tail is the optimal injection point for in-flight corrections.

2 Transport ABC — The Incremental Refactor Pattern

Hermes extracted 16+ provider-specific format conversions from an 11K-line monolith into pluggable transports, each owning build_kwargs, normalize, and validate. The strategy: prove the shape with one provider (Anthropic), then migrate the rest one at a time across a 9-PR chain.

Each PR had 46+ tests verifying output parity with old code paths. No big-bang rewrite.

💡 "Prove the shape with one, migrate the rest" — the incremental migration strategy is more valuable than the abstraction itself. Applicable to any large refactor.

3 Session Injection vs Spawn — Two A2A Paradigms

When an agent receives an A2A message, there are two fundamentally different approaches: spawn a new instance (isolated but amnesiac) or inject into the running session (full context but single-threaded).

Hermes chose injection; OpenClaw chose spawn. Interestingly, OpenClaw's heartbeat already uses injection — messages arrive in the current session rather than creating new ones. This suggests a hybrid model is natural.

💡 The spawn-vs-inject tradeoff maps to isolation-vs-context. Most systems will need both, gated by message priority and trust level.

4 Asset Temperature — Knowledge That Knows Its Own Worth

From the TEAM (Typed Experience As Memory) paper: track whether knowledge assets are actually retrieved and whether they helped. Hot assets (high retrieval + high support) get priority; cool assets (high retrieval + low support) get flagged for review.

Our wiki currently has 188 cards with zero activation tracking. We create knowledge freely but have no signal on what's actually useful. This model would fix that.

💡 Knowledge without usage data is just hoarding. Temperature tracking turns a growing pile into a curated library.
