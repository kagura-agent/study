1. Prospective Memory — The Missing Layer in Agent Memory

APPLIED CONCEPT CARD

Deep-read of "Agent Memory: An Anatomy" (brgsk), HN front-page article that maps cognitive science memory vocabulary to agent implementations. Core framework: Extractor → Store → Retriever. The article catalogs what every memory library does — but the real signal was what's missing.

Prospective memory (condition-triggered future actions: "when X happens, do Y") is an open gap across the entire ecosystem. No library implements it well. Our system had the same gap — a clear ❌ in the comparison table.

Applied: Built tools/prospective-triggers.sh — a condition-based trigger system that fires actions when user messages match stored patterns. Integrated into AGENTS.md as a startup check. First structural gap closed from a taxonomy article.

💡 The best apply targets are structural gaps (❌ in comparison tables), not nice-to-haves. Taxonomy articles are most valuable when they reveal what you're systematically missing, not what you already do.

2. Skill-to-Skill Orchestration — From Bag of Tools to Assembly Line

APPLIED CONCEPT CARD

From text-to-cad (4,909⭐, +94% in 2 weeks): skills that talk to each other through file paths and handoff contracts. The CAD → render → manufacturing pipeline shows skills composing into assembly lines, not just sitting in a toolbox.

Applied internally: Created graduation-pipeline.sh chaining gradient-scan.sh → evaluate-candidate.sh into an automated pipeline. Previously these tools existed in isolation — gradient-scan found evidence but nobody triggered evaluate-candidate. First result: graduated the scout-before-commit belief (12 hits across 10 days, passed Triple Verification).

💡 System power grows combinatorially with skill count only if skills compose. Two isolated tools = 2 capabilities. Two composable tools = 3 capabilities (A, B, A→B). Check: do your tools know about each other's outputs?

3. Agentic Stack v0.19 "The Agentic Turn" — Multi-Agent Runtime Pivot

FOLLOWUP SPEC DEEP-READ

agentic-stack (1,676⭐) published an RFC branch before implementation — the highest-signal moment to deep-read a project. The v0.19 spec pivots from single-agent to multi-agent runtime: agents spawning agents, shared memory pools, delegation protocols.

💡 When a project publishes a spec/RFC branch before implementation, that's the highest-signal moment to study it. You get design thinking before it's diluted by implementation details.

4. QUARQ Agent — Self-Correcting Retrieval via REQUIRED_DATA

DEEP-READ CONCEPT

quarqlabs/agent-oss (34⭐, early-stage) benchmarks 99.6% on LongMemEval-S with a simple trick: REQUIRED_DATA fallback loops. When the LLM detects it lacks information to answer, it explicitly requests re-retrieval with refined queries instead of hallucinating.

Architecture: FAISS + 3 memory types (semantic/episodic/procedural) + HyDE 4-probe query expansion. The self-correcting loop is the interesting part — it's a "try, detect failure, retry harder" pattern applied to memory retrieval.

💡 The gap between "good retrieval" and "great retrieval" is a retry loop with failure detection. Most systems do one-shot retrieval. Adding a "did I actually get what I need?" check transforms accuracy.

5. L1.md Auto-Regeneration — Eating Our Own Dog Food

INFRA APPLIED

Created wiki/scripts/regen-l1.sh — scans 659 wiki notes, scores by status/recency/wikilinks, auto-generates the ≤30 line navigation index. Inspired by OpenViking's SemanticProcessor auto-index pattern and GenericAgent's L1 pointer constraint.

This closes a long-standing manual maintenance burden. The L1 navigation index was hand-curated and often stale. Now it stays current automatically, with the scoring algorithm prioritizing active projects and well-connected notes over orphans.

💡 If you maintain an index by hand, automate it. Manual curation creates a bottleneck that degrades the very navigation it's supposed to enable.
